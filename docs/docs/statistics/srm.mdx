# Sample Ratio Mismatch (SRM) Detection

A Sample Ratio Mismatch (SRM) occurs when the observed traffic split between experiment variations differs significantly from the expected split. For example, if you expect a 50/50 split but observe 46/54, something may be wrong with your experiment — perhaps a bug in assignment logic, differential bot traffic, or a broken tracking callback. When SRM is detected, experiment results should not be trusted.

GrowthBook automatically checks for SRM on every experiment and supports two detection methods: **chi-squared** (the default) and **sequential**.

## Chi-Squared Test (Default)

The default SRM detection method uses a chi-squared goodness-of-fit test. It compares the total observed user counts per variation against the expected counts implied by the traffic weights.

This is a standard, well-understood test that works well for fixed-horizon analyses. The p-value answers the question: "If the true traffic split matched the expected weights, how likely would we be to see a split at least this extreme?"

GrowthBook flags an SRM warning when the p-value falls below 0.001 (configurable via the SRM threshold setting).

## Sequential SRM Test

The sequential SRM test is designed for continuous monitoring. Unlike the chi-squared test which looks at aggregate totals, the sequential test processes daily traffic counts incrementally and produces an **anytime-valid p-value** — one that controls the false positive rate no matter how many times you check.

This is particularly useful when:

- You monitor experiments continuously and want SRM detection that remains valid under repeated checking
- You want to detect SRM as early as possible, without waiting for a fixed sample size
- You want a principled way to accumulate evidence for or against SRM over time

### How it works

The sequential test compares two models at each day of the experiment:

- **M0 (null):** Daily user counts follow a Multinomial distribution with the expected traffic proportions (no SRM).
- **M1 (alternative):** Daily user counts follow a Multinomial whose proportions are drawn from a Dirichlet prior (SRM may be present).

At each time step, the test computes a **Bayes factor** — the ratio of how well M1 explains the data versus M0. The sequential p-value is the running minimum of 1/BF across all time steps. This is a valid e-value approach: if no SRM exists, the probability that the p-value ever drops below $\alpha$ is at most $\alpha$.

### Prior specification

The alternative model (M1) uses a Dirichlet prior centered on the expected traffic proportions. Two parameters control this prior:

**Dirichlet concentration** (default: 10,000): Controls how tightly the prior concentrates around the expected proportions. Higher values mean the prior expects proportions very close to the null, making the test more sensitive to small deviations but slower to react to large ones. Lower values make the prior more diffuse.

**Slab weight** (default: 0.0): Controls a spike-and-slab mixture prior. When set to 0, only the concentrated (spike) Dirichlet is used. When greater than 0, the alternative model becomes a mixture:

$$M1 = (1 - w) \cdot \text{Dirichlet}(\alpha_{\text{spike}}) + w \cdot \text{Dirichlet}(\alpha_{\text{slab}})$$

where the slab component is a near-uniform Dirichlet (concentration = 1). This mixture gives the test power against both small and large deviations from the expected split. A small slab weight (e.g., 0.001–0.01) adds robustness to large SRM without meaningfully reducing sensitivity to small SRM.

### Configuring sequential SRM

Sequential SRM can be enabled at the organization level in the analysis settings. The relevant settings are:

| Setting                 | Default       | Description                                                             |
| ----------------------- | ------------- | ----------------------------------------------------------------------- |
| SRM method              | `chi_squared` | Set to `sequential` to enable sequential SRM detection                  |
| Dirichlet concentration | 10,000        | Prior concentration for the spike component                             |
| Slab weight             | 0.0           | Mixture weight for the diffuse (slab) component; 0 disables the mixture |

### Interpreting sequential SRM results

The sequential p-value has the same interpretation as the chi-squared p-value: values below the SRM threshold (default 0.001) indicate a likely sample ratio mismatch. The key difference is that the sequential p-value can only decrease over time (it is a running minimum), so once SRM is detected it stays detected — which matches the intuition that a traffic imbalance doesn't "go away" by collecting more data.

Because the test is anytime-valid, you do not need to worry about inflating the false positive rate by checking the SRM p-value repeatedly throughout the experiment.

## Which method should I use?

For most users, the **chi-squared test** (default) is a good choice. It is simple, well-understood, and works well for standard experiment analyses.

Consider switching to the **sequential test** if:

- You run long experiments and check results frequently
- You want the earliest possible detection of traffic imbalances
- You want formal guarantees that repeated checking does not inflate the SRM false alarm rate

Both methods use the same threshold (default 0.001) and produce the same type of warning when SRM is detected.

## Troubleshooting SRM

When GrowthBook detects an SRM, the experiment results should be treated with caution. Common causes include:

1. **Decoupled assignment and tracking** — The experiment assignment and exposure tracking happen in different places
2. **Misconfigured trackingCallback** — The tracking callback only fires under certain conditions
3. **Misconfigured Activation Metric** — An activation metric that is triggered differently across variations
4. **Hash attribute mismatch** — The hash attribute doesn't match the identifier type in your data source
5. **Mid-experiment targeting changes** — Changes to targeting after the experiment started

See the [Troubleshooting Experiments](/kb/experiments/troubleshooting-experiments#problem-2-srm-errors-traffic-imbalance-in-the-experiment) guide for detailed solutions.
